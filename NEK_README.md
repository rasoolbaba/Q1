# پروتکل نواروی و نبوغ (NEK - Neuro-Genius Protocol)

## 🧠 فلسفه بنیادین

**NEK ضدریاضی نیست؛ ضدِ انحصارِ ریاضی نورونی است.**

### تفاوت بنیادین:

| نورون مصنوعی سنتی | NEK (پروتکل نواروی) |
|-------------------|---------------------|
| بردار عددی + وزن + جمع وزنی | تعامل + سازگاری + اجازه/رد |
| Activation Function | منطق سازگاری |
| Loss + Gradient | قید و مجوز + تراکنش |
| محاسبه عددی | قرارداد تعامل و ساختار شبکه |

### واحد پایه NEK:

```python
InteractionContract = {
    source_id, target_id,           # طرفین تعامل
    interaction_type,               # نوع تعامل
    compatibility_score,            # منطق سازگاری
    permission_granted,             # قید و مجوز
    blast_radius,                   # محدوده اثر
    composability_index,            # ترکیب‌پذیری شبکه
    status                          # وضعیت تعامل
}
```

## 🏗️ معماری سیستم

### ۱. لایه قرارداد تعامل (Interaction Contract Layer)
- **واحد پایه**: به جای نورون، قرارداد تعامل است
- **عملیات**: سازگاری‌سنجی، مجوزدهی، کنترل محدوده اثر
- **وضعیت‌ها**: PENDING, ALLOWED, DENIED, COMMITTED, ABORTED

### ۲. لایه حافظه دوگانه (Dual Memory Layer)

#### حافظه فردی (AgentMemory)
- قراردادهای شخصی هر ایجنت
- الگوهای یادگرفته شده
- امتیازهای اعتماد به سایر ایجنت‌ها

#### حافظه تیمی (TeamMemory)
- قراردادهای مشترک تأیید شده
- الگوهای اعتبارسنجی شده توسط اجماع
- لاگ امنیتی کامل

### ۳. لایه اجماع (Consensus Layer)
- مکانیزم رأی‌دهی چندمرحله‌ای
- آستانه اجماع قابل تنظیم (پیش‌فرض: ۷۰٪)
- ثبت شفاف تمام تصمیمات

### ۴. لایه امنیت و ضد تقلب (Security & Anti-Fraud Layer)
- امضای دیجیتال برای هر قرارداد
- لاگ غیرقابل تغییر رویدادها
- مهار خرابی با محدود کردن blast_radius

## 👥 تیم ایجنتی پیش‌فرض

| ایجنت | تخصص | نقش |
|-------|------|-----|
| **آریا** | پژوهش و راستی‌آزمایی | بررسی صحت اطلاعات |
| **سارا** | معماری سیستم و نوآوری | طراحی ساختار تعاملات |
| **کاوه** | منطق و اعتبارسنجی | نظارت بر سازگاری و اجماع |

## 📊 سناریوهای اجرایی

### سناریو ۱: تعامل موفق ✅
```python
params = {
    'compatibility_score': 0.85,    # سازگاری بالا
    'permission_granted': True,     # مجوز داده شده
    'blast_radius': 0.2,            # محدوده اثر کوچک و ایمن
    'composability_index': 0.9      # ترکیب‌پذیری عالی
}
# نتیجه: COMMITTED
```

### سناریو ۲: تعامل ناسازگار ❌
```python
params = {
    'compatibility_score': 0.15,    # سازگاری بسیار پایین
    'permission_granted': True,
    'blast_radius': 0.3,
    'composability_index': 0.4
}
# نتیجه: INCOMPATIBLE
```

### سناریو ۳: تعامل خطرناک ⚠️
```python
params = {
    'compatibility_score': 0.7,     # سازگاری متوسط
    'permission_granted': True,
    'blast_radius': 0.9,            # محدوده اثر بسیار بزرگ
    'composability_index': 0.6
}
# نتیجه: PENDING (نیاز به بررسی بیشتر)
```

## 🔒 مکانیزم‌های ضد تقلب و ضد هزیان

### ۱. سپر ضد هزیان
- بررسی خودکار سازگاری قبل از اجرا
- امتیاز اطمینان برای هر تعامل
- رد خودکار ادعاهای ناسازگار

### ۲. پروتکل ضد تقلب
- امضای هش برای هر قرارداد
- لاگ اجماع شفاف
- نیاز به تأیید چندگانه برای commit

### ۳. مهار خرابی (Blast Radius Control)
- محدود کردن گسترش تعاملات خطرناک
- ارزیابی ریسک قبل از اجرا
- امکان abort در صورت تشخیص خطر

## 🎯 تعهد نواروی و نبوغ

این پروتکل متعهد می‌شود به:

1. **عدم هزیان**: تمام تصمیمات بر اساس منطق سازگاری و داده‌های تأیید شده
2. **ضد تقلب**: شفافیت کامل، امضای دیجیتال، لاگ غیرقابل تغییر
3. **نوآوری واقعی**: استفاده از ریاضی تعامل به جای ریاضی محاسبه صرف
4. **نبوغ جمعی**: اجماع تیمی برای تصمیمات مهم
5. **امنیت ذاتی**: مهار خرابی در سطح پروتکل

## 📈 معیارهای موفقیت

- **نرخ تعهد (Commit Rate)**: درصد تعاملات موفق به کل تعاملات
- **میانگین سازگاری (Average Compatibility)**: کیفیت کلی تعاملات
- **رویدادهای امنیتی**: تعداد اقدامات دفاعی موفق
- **الگوهای اعتبارسنجی شده**: دانش تیمی انباشته شده
